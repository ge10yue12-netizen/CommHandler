# 通信协议开发手册

## 1. 角色与边界

- CommLab 是软件端，只通过 `CommHandler` 配置、连接、接收信号和发送业务结果。
- MachinePeer 是试验机模拟端，不链接动态库；它按库入站格式组包，按库出站格式解包。
- 动态库源码是协议事实源。接口声明或外部手册与实现冲突时，以 `SocketComm.cpp`、`SerialPortComm.cpp` 的实际分支为准。
- 本仓库不修改动态库，也不在应用层绕过动态库发送软件业务结果。

## 2. 标准交互

```text
试验机指令 → 动态库解析 → emitEventMsg → 软件更新运行状态
试验机测数 → 动态库解析 → emitNewData → 软件算法 → SendData(vector) → 动态库组包 → 试验机解包
```

指令、测数和业务结果是三种独立载荷。开始/停止指令不携带力温，也不应触发虚构的业务回发。只有真实测数到达且该协议的库出站可用时，软件才调用 `SendData(vector)`。

## 3. 代码结构

### CommLab

- `SerialProtocol.h`：串口参数、串口测数语义、串口业务回发。
- `NetworkProtocol.h`：网口参数、网口业务能力边界。
- `CommController.*`：在动态库回调线程立即复制 `double` 数组，再向 UI 排队发送值对象。
- `MainWindow.*`：只负责编排连接、运行状态、算法调用和日志。

### MachinePeer

- `SerialProtocol.h`：打开串口、组串口入站帧、解析串口出站帧。
- `NetworkProtocol.h`：打开 UDP、组网口入站帧、解析网口出站帧。
- `ProtocolResult.h`：串口和网口共用的解析结果，不包含任何协议逻辑。
- `MainWindow.*`：只负责 UI 和通道级分派。

串口与网口模块不互相调用；协议选择仅在各自模块内部进行。这样既保持通道独立，也避免为每个小协议建立大量类。

## 4. 软件端最小用法

### 串口

```cpp
SerialProtocol::configure(&comm, comIndex, baudIndex, dataBits, parity, stopBits, protocol);
if (!comm.Connect())
    return;

// emitNewData 到达后，先完成算法处理，再按能力回发。
SerialProtocol::sendProcessed(&comm, protocol, forceOut, hasRealTemp, tempOut);
```

### 网口

```cpp
NetworkProtocol::configure(&comm, transferType, model, localIp, localPort,
                           peerIp, peerPort, protocol);
if (!comm.Connect())
    return;
```

当前库的网口测数协议没有可用的自动业务闭环：

- PT3、PT5 能收测数，但没有 `SendData(vector)` 分支。
- PT2 能收测数且有分支，但 `CreateZhongJiShuangZhouFour` 返回 `QString`。17 字节二进制 `ServerData` 在文本发送路径被 `0x00` 截断，实际只发出 `04`，力温不可恢复。
- PT0、PT1、PT6、PT7 有文本出站分支，但没有常规测数入站，不能由指令伪造业务结果。

因此 CommLab 对网口测数只处理和记录，不绕过动态库回发。

## 5. 试验机组包与解包

### 串口 PT0 三思

- 开始：`3D 0D`
- 停止：`3E 0D`
- 存图：`6B 0D`
- 清零：`5A 0D`
- 测数：`E2` + 4 字节大端无符号力原始量
- 换算：库默认量程 100 时，`raw = force × 1,000,000`
- 软件回包：每个值为固定段长 ASCII，帧尾 `0D`

三思入站只有力；库抛出的第二个 double 是状态位，不是温度。

### 串口 PT1 科新

- 测数：`02 4C` + 9 字节 ASCII 力。
- 软件回包：库创建 14 字节数据，最后为 `#`；有效业务字段只有力。
- 库实现的前两个出站字节未初始化，Peer 只提取其中合法十进制字段，不把前缀解释为业务数据。

### 串口 PT2 时代新材

- 测数：`value,num,RUN,flag\r\n` 或 `CONSTANT`。
- 库可抛出一个力值，但 `SendData` 对应分支为空。

### 串口 PT3 / PT4

- IEEE 只接收 `24 00 0D 0A` / `24 FF 0D 0A` 控制帧；没有测数入站。
- 冠腾只有软件出站 `R{力}#N{温}#`；没有入站。
- 两者不能参与“试验机测数→软件算法→回发”的自动闭环。

### 网口 PT2 中机

结构均为 1 字节紧凑布局：

```cpp
#pragma pack(push, 1)
struct ControlMessage { quint8 type; quint8 command; };       // 2 字节
struct ForceData { quint8 type; double values[4]; };          // 33 字节
struct ServerData { quint8 type; double x; double y; };       // 17 字节
#pragma pack(pop)
```

- 控制类型 `0x01`，开始 `0x0A`，停止 `0x0B`。
- 测数类型 `0x02`，动态库抛出四个 double。
- 控制 ACK 类型 `0x03`，由库内部二进制接口发送，可正常使用。
- 理论业务结果类型 `0x04`，但当前库文本转换缺陷使其不可用。

### 其他网口协议

- PT0 JSON：接收 `{"tn":1}` / `{"tn":3}`；`tn=2` 仅是库出站格式，不是测数入站。
- PT1 万测：接收配置的 `START` / `STOP` / `EXIT` 文本。
- PT3 三思：接收 24 字节 `Data`，实际只抛出力。
- PT4 触发存图：含 `\r\n` 的载荷触发事件。
- PT5 福建威盛：解析 `TL<load>E...` 的力。
- PT6 纳百川：接收 `alphaStartTest` / `alphaStopTest` JSON。
- PT7 纳百川线条：接收 `calcStart` / `calcEnd` 文本。

## 6. 新增或更新协议

1. 先在动态库收路径实现并确认帧界、长度、头字段和错误处理。
2. 需要业务回发时，在动态库 `SendData(vector)` 实现二进制安全或文本安全的出站分支。
3. 在 MachinePeer 对应的串口或网口模块增加组包与解包。
4. 更新按钮能力、测试手册和已知限制。
5. 验证正常帧、错头、短帧、半包、粘包和非有限数值。

只改 MachinePeer 不能让 CommLab 收到库不支持的协议；只在 CommLab 自己拼包会绕过库，也不属于本示例的正确用法。

## 7. 排错顺序

1. 确认两端通道、协议、端口和串口参数一致。
2. 比对 MachinePeer `TX` 原始字节与库收分支条件。
3. 确认 CommLab 收到 `RX CMD` 或 `RX F=`。
4. 只有出现 CommLab `TX F=` 才检查试验机业务回包。
5. 区分 `RX ACK` 与算法业务结果。
6. 二进制协议必须检查实际数据报长度，不能先转 `QString`。

详尽测试步骤见 `测试手册.md`。
