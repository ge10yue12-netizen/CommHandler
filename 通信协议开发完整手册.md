# CommHandler 通信动态库完整开发与使用手册

## 1. 角色与边界

- CommLab 是软件端，只通过 `CommHandler` 配置、连接、接收信号和发送业务结果。
- MachinePeer 是试验机模拟端，不链接动态库；它按库入站格式组包，按库出站格式解包。
- 动态库源码是协议事实源。接口声明或外部手册与实现冲突时，以 `SocketComm.cpp`、`SerialPortComm.cpp` 的实际分支为准。
- 本仓库不修改动态库，也不在应用层绕过动态库发送软件业务结果。

## 2. 标准交互

```text
试验机指令 → 动态库解析 → emitEventMsg → 软件更新运行状态
试验机测数 → 动态库解析 → emitNewData → 软件算法 → SendData(vector) → 动态库组包 → 试验机解包
```

指令、测数和业务结果是三种独立载荷。开始/停止指令不携带力温，也不应触发虚构的业务回发。只有真实测数到达且该协议的库出站可用时，软件才调用 `SendData(vector)`。

## 3. 代码结构

### CommLab

- `SerialProtocol.h`：串口参数、串口测数语义、串口业务回发。
- `NetworkProtocol.h`：网口参数、网口业务能力边界。
- `CommController.*`：在动态库回调线程立即复制 `double` 数组，再向 UI 排队发送值对象。
- `MainWindow.*`：只负责编排连接、运行状态、算法调用和日志。

### MachinePeer

- `SerialProtocol.h`：打开串口、组串口入站帧、解析串口出站帧。
- `NetworkProtocol.h`：打开 UDP、组网口入站帧、解析网口出站帧。
- `ProtocolResult.h`：串口和网口共用的解析结果，不包含任何协议逻辑。
- `MainWindow.*`：只负责 UI 和通道级分派。

串口与网口模块不互相调用；协议选择仅在各自模块内部进行。这样既保持通道独立，也避免为每个小协议建立大量类。

## 4. 软件端最小用法

### 串口

```cpp
SerialProtocol::configure(&comm, comIndex, baudIndex, dataBits, parity, stopBits, protocol);
if (!comm.Connect())
    return;

// emitNewData 到达后，先完成算法处理，再按能力回发。
SerialProtocol::sendProcessed(&comm, protocol, forceOut, hasRealTemp, tempOut);
```

### 网口

```cpp
NetworkProtocol::configure(&comm, transferType, model, localIp, localPort,
                           peerIp, peerPort, protocol);
if (!comm.Connect())
    return;
```

当前库的网口测数协议没有可用的自动业务闭环：

- PT3、PT5 能收测数，但没有 `SendData(vector)` 分支。
- PT2 能收测数且有分支，但 `CreateZhongJiShuangZhouFour` 返回 `QString`。17 字节二进制 `ServerData` 在文本发送路径被 `0x00` 截断，实际只发出 `04`，力温不可恢复。
- PT0、PT1、PT6、PT7 有文本出站分支，但没有常规测数入站，不能由指令伪造业务结果。

因此 CommLab 对网口测数只处理和记录，不绕过动态库回发。

## 5. 试验机组包与解包

### 串口 PT0 三思

- 开始：`3D 0D`
- 停止：`3E 0D`
- 存图：`6B 0D`
- 清零：`5A 0D`
- 测数：`E2` + 4 字节大端无符号力原始量
- 换算：库默认量程 100 时，`raw = force × 1,000,000`
- 软件回包：每个值为固定段长 ASCII，帧尾 `0D`

三思入站只有力；库抛出的第二个 double 是状态位，不是温度。

### 串口 PT1 科新

- 测数：`02 4C` + 9 字节 ASCII 力。
- 软件回包：库创建 14 字节数据，最后为 `#`；有效业务字段只有力。
- 库实现的前两个出站字节未初始化，Peer 只提取其中合法十进制字段，不把前缀解释为业务数据。

### 串口 PT2 时代新材

- 测数：`value,num,RUN,flag\r\n` 或 `CONSTANT`。
- 库可抛出一个力值，但 `SendData` 对应分支为空。

### 串口 PT3 / PT4

- IEEE 只接收 `24 00 0D 0A` / `24 FF 0D 0A` 控制帧；没有测数入站。
- 冠腾只有软件出站 `R{力}#N{温}#`；没有入站。
- 两者不能参与“试验机测数→软件算法→回发”的自动闭环。

### 网口 PT2 中机

结构均为 1 字节紧凑布局：

```cpp
#pragma pack(push, 1)
struct ControlMessage { quint8 type; quint8 command; };       // 2 字节
struct ForceData { quint8 type; double values[4]; };          // 33 字节
struct ServerData { quint8 type; double x; double y; };       // 17 字节
#pragma pack(pop)
```

- 控制类型 `0x01`，开始 `0x0A`，停止 `0x0B`。
- 测数类型 `0x02`，动态库抛出四个 double。
- 控制 ACK 类型 `0x03`，由库内部二进制接口发送，可正常使用。
- 理论业务结果类型 `0x04`，但当前库文本转换缺陷使其不可用。

### 其他网口协议

- PT0 JSON：接收 `{"tn":1}` / `{"tn":3}`；`tn=2` 仅是库出站格式，不是测数入站。
- PT1 万测：接收配置的 `START` / `STOP` / `EXIT` 文本。
- PT3 三思：接收 24 字节 `Data`，实际只抛出力。
- PT4 触发存图：含 `\r\n` 的载荷触发事件。
- PT5 福建威盛：解析 `TL<load>E...` 的力。
- PT6 纳百川：接收 `alphaStartTest` / `alphaStopTest` JSON。
- PT7 纳百川线条：接收 `calcStart` / `calcEnd` 文本。

## 6. 新增或更新协议

1. 先在动态库收路径实现并确认帧界、长度、头字段和错误处理。
2. 需要业务回发时，在动态库 `SendData(vector)` 实现二进制安全或文本安全的出站分支。
3. 在 MachinePeer 对应的串口或网口模块增加组包与解包。
4. 更新按钮能力、测试手册和已知限制。
5. 验证正常帧、错头、短帧、半包、粘包和非有限数值。

只改 MachinePeer 不能让 CommLab 收到库不支持的协议；只在 CommLab 自己拼包会绕过库，也不属于本示例的正确用法。

## 7. 排错顺序

1. 确认两端通道、协议、端口和串口参数一致。
2. 比对 MachinePeer `TX` 原始字节与库收分支条件。
3. 确认 CommLab 收到 `RX CMD` 或 `RX F=`。
4. 只有出现 CommLab `TX F=` 才检查试验机业务回包。
5. 区分 `RX ACK` 与算法业务结果。
6. 二进制协议必须检查实际数据报长度，不能先转 `QString`。

详尽测试步骤见 `测试手册.md`。

## 9. 动态库总体架构

### 9.1 四个通信模块

`CommHandler` 构造时一次性创建四个模块：

```7:16:CommHandler/CommHandler.cpp
CommHandler::CommHandler(QObject* parent)
	: QObject(parent)
	, m_curCommModule(nullptr)
{
	m_commModules[NETWORK] = new SocketComm();
	m_commModules[SERIAL] = new SerialPortComm();
	m_commModules[VOLTAGE] = new VoltageComm();
	m_commModules[PULSE] = new PulseComm();
	m_curCommModule = m_commModules.value(NETWORK, nullptr);
	ConnectSlots();
}
```

`CommType` 定义在 `CommHandler/Common/UIDef.h:17-23`：

- `NETWORK = 0`
- `SERIAL = 1`
- `VOLTAGE = 2`
- `PULSE = 3`

`SetCommType()` 只切换“当前配置和连接模块”；`SendData(vector, type)` 则直接按传入的 `CommType` 查找模块，并不依赖当前模块。

### 9.2 门面调用链

```text
应用构造 CommHandler
  └─ 创建 SocketComm / SerialPortComm / VoltageComm / PulseComm
      └─ 各模块注册参数 getter/setter 并初始化默认值

应用 SetCommType(type)
  └─ m_curCommModule = m_commModules[type]

应用 setParameter / Connect
  └─ 调用当前模块

设备数据到达
  └─ 子模块解析
      ├─ emitNewData：测量数据
      ├─ emitEventMsg：开始、停止、存图等控制事件
      └─ emitEventMsgAndData：带标距等附加参数的事件

应用完成算法
  └─ SendData(vector, type)
      └─ 指定模块组包并发送
```

### 9.3 公开 API 的真实分派

- `SetSysParInfo`、`GetSysParInfo`、`Connect`、`SaveParInfo`、`LoadParInfo`：当前模块。
- `SendData(vector, CommType)`、`SendData(QString, CommType)`：参数指定的模块。
- `HardTrigger`、`ReleaseHardTrigger`、`ReadCIFreq`、`ReadADData`：固定转发到电压模块。
- `SendCaliData`：固定转发到脉冲模块。
- `SendRequireMsg`：当前实现恒返回 `false`。
- `setForceUnit`：当前实现为空。

### 9.4 信号内存所有权

动态库多个解析分支会执行以下模式：

```cpp
double* data = new double[count];
emit emitNewData(data, count, type);
delete[] data;
```

因此：

1. `size` 表示 `double` 的数量，不是字节数。
2. `void*` 仅在信号同步调用期间有效。
3. 禁止把该裸指针保存到队列或 UI 线程后再读取。
4. 应使用 `Qt::DirectConnection` 在发出线程立即复制，再把值对象排队给 UI。

CommLab 已在 `CommLab/src/CommController.cpp:21-28` 正确实现：

```cpp
// 在动态库回调线程中立即复制临时数组。
QObject::connect(&m_comm, &CommHandler::emitNewData, this,
	[this](void* data, int size, int type)
	{
		if (!data || size <= 0 || size > kMaxCallbackDoubles)
			return;
		QVector<double> copied(size);
		std::memcpy(copied.data(), data, sizeof(double) * size);
		emit safeDataReceived(copied, type);
	}, Qt::DirectConnection);
```

第三方项目必须采用同样的所有权策略。

## 10. 参数系统与生命周期

### 10.1 推荐调用顺序

```text
构造 QApplication
  → 构造 CommHandler
  → 连接 emitNewData / emitEventMsg
  → SetCommType
  → setParameter 或 SetSysParInfo
  → Connect
  → 收数、处理、按能力 SendData
  → Disconnect
  → 析构 CommHandler
```

子模块构造函数已经调用自己的 `ConnectSlots()`；应用不需要再次调用公开的 `CommHandler::ConnectSlots()`。

### 10.2 setParameter 的使用规则

推荐先调用 `SetCommType()`，再设置该模块参数。字符串键拼错会抛 `std::invalid_argument`；模板版 `CommHandler::setParameter` 会尝试其他模块，全部失败后抛 `std::runtime_error`。

网口常用键：

```text
iTransferType, iModel, sLocalIP, iLocalPort,
sDestIP, iDestPort, iProtoType, iSymbolType,
bOpenCMD, cCMD, bOnlineCollect, bAllOpenCamera,
bInquireSendFlag, input, output
```

其中 `sDestIP`、`iDestPort` 在库内映射到 `SocketInfo::sPurIP`、`iPurPort`。

串口常用键：

```text
iComPort, iComBaud, iParity, iDataBits, iStopBits,
iProtocolType, dInForceRange, bInquireSendFlag,
input, output
```

### 10.3 Connect/Disconnect 注意事项

- 串口 `iComPort=0` 表示 `COM1`。
- 串口波特率索引：0/1/2/3/4/5 对应 4800/9600/19200/38400/57600/115200。
- 网口 `iTransferType=0` 为 TCP，`1` 为 UDP。
- 网口 `iModel=0` 为服务端，`1` 为客户端。
- 电压 `Connect(type)`：0=AD、1=DA、2=CI、3=DI。
- `CommHandler::Disconnect(-1)` 的循环内提前 `return`，当前只能断开第一个模块；不要依赖其“关闭全部”语义。
- 串口 `Connect()` 没有把 `bComOpened` 置为 `true`，重复连接保护并不可靠。

## 11. 串口协议源码详解

串口入口：

- 打开：`SerialPortComm.cpp:76-137`
- 业务发送：`SerialPortComm.cpp:157-253`
- 数据换算：`SerialPortComm.cpp:408-470`
- 接收分发：`SerialPortComm.cpp:1527-1915`

### 11.1 PT0 三思测控

#### 控制入站

必须启用 `input.iTriggerCtrl`。动态库对一次 `readAll()` 的完整十六进制字符串做精确匹配：

- `3D 0D`：开始
- `3E 0D`：停止
- `6B 0D`：自动存图
- `5A 0D`：清零

控制分支位于 `SerialPortComm.cpp:1542-1565`。源码中的 `SendOKResponse()` 均已注释，当前不会自动返回 `OK 0D`。

#### 标距入站

以字节 `0x47`（ASCII `G`）开始，可包含：

```text
G H +<横向标距> V +<纵向标距> CR
```

库把结果放入 `QVariantMap` 的 `H`、`V` 字段，并发出 `W_CUSTOM_COMM_RESET_LVE_LENGTH`。实现位于 `SerialPortComm.cpp:1567-1639`。

#### 测数入站

帧格式：

```text
E2 + raw_force:uint32_be
```

换算公式：

```text
force = raw / 1,000,000 / 100 × dInForceRange
```

默认量程 `dInForceRange=100` 时：

```text
raw = force × 1,000,000
```

例：`force=1.0`：

```text
raw = 1,000,000 = 0x000F4240
frame = E2 00 0F 42 40
```

库抛出两个 `double`：

- `[0]`：力
- `[1]`：`COMDATA_STATUS_ENABLE` 状态值，不是温度

#### 半包与粘包

库从本次数据末尾向前找最后一个 `E2`：

- 后续不足 4 字节时，保存到 `m_sInputData`。
- 下一次没有新帧头时，追加数据。
- 超过 4 字节时只取前 4 字节。
- 多帧粘包只保留最后一个 `E2` 对应值。

这不是完整的多帧状态机；需要全部样本时，发送端应避免把多帧一次性粘连。

#### 业务出站

每个值占 `m_iDataBits` 字节。正常 8 数据位配置下，每段为 8 字节：

```text
[+|-][ASCII 数值，共占 8 字节]
```

多通道依次拼接，最后追加 `0D`。组包函数：

- `SerialPortComm::FloatToHex`：`978-1038`
- `SerialPortComm::SendData case 0`：`174-186`

CommLab 当前只回发真实入站力，不伪造温度。

### 11.2 PT1 长春科新

#### 测数入站

```text
02 4C + 9 字节 ASCII 力值
```

库搜索 `024C`，取其后 18 个十六进制字符，再由 `HexToFloat()` 把 ASCII 的符号、数字和小数点转换为 `double`。

输出同样为两个 `double`：力和状态位。

#### 半包缺陷

`SerialPortComm.cpp:1817-1826` 的半包续传路径错误地搜索三思帧头 `E2`，而不是科新帧头 `024C`。因此科新测数应尽量在一次串口写操作中完整发送。

控制半包路径还存在把 `m_sInputData` 与 `m_sCtrlData` 混用的问题。

#### 业务出站

理论长度 14 字节：

```text
[未初始化 2 字节][符号/5位整数/小数点/5位小数][#]
```

`FixedDoubleToHex()` 只写入索引 2 至 13；索引 0、1 未初始化。MachinePeer 使用正则只提取有效十进制字段，不把前两个随机字节解释为业务内容。

### 11.3 PT2 时代新材

帧界为 `\r\n`，库使用 `g_serialBuffer` 正确处理半包和多行粘包。

入站格式：

```text
value,num,RUN,flag\r\n
```

第三段也允许 `CONSTANT`。库只读取第一段 `value` 并抛出一个 `double`。当前信号 `type=1`，但实际数据是力，不应按温度解释。

`SendData case 2` 为空，因此只能接收，不能业务回发。

### 11.4 PT3 IEEE

控制入站：

- `24 00 0D 0A`：开始
- `24 FF 0D 0A`：停止

当前没有测数入站路径。

业务出站要求至少 5 个值：

```text
vData[0..3] = strain1, strain2, movement1, movement2
vData[4]    = time
reserve     = 0
```

56 字节帧：

```text
offset 0..1   : 24 01
offset 2..49  : 6 个 float 的小端 IEEE754 字节转成十六进制 ASCII
offset 50..53 : 24 个 float 原始字节累加和的低 16 位，转十六进制 ASCII
offset 54..55 : 0D 0A
```

实现位置：

- 原始 float 拷贝及校验和：`SerialPortComm.cpp:785-835`
- 帧封装：`877-965`
- 字节转十六进制：`1256-1331`

由于没有测数入站，CommLab 不应把开始指令当作回发触发。

### 11.5 PT4 冠腾

只有业务出站：

```text
R{力}#N{温}#
```

数值格式为符号加固定 4 位小数，例如：

```text
R+001.2345#N+025.0000#
```

实际宽度由 `QString::arg(absVal, 8, 'f', 4, '0')` 生成。调用方必须至少传两个值，否则 `vData.at(1)` 会越界。

### 11.6 串口共用门控

所有 `SendData(vector)` 首先检查 `bInquireSendFlag`。轮询模式发送完成后会自动将其清零。

源码尾部还识别 `3F 0D` 并把 `bInquireSendFlag` 置为 `true`，但 PT2、PT3 分支提前 `return`，不会走到这段共用逻辑。

## 12. 网口协议源码详解

网口入口：

- 连接：`SocketComm.cpp:82-108`
- 业务发送：`SocketComm.cpp:130-189`
- 组包：`SocketComm.cpp:490-598`
- 接收分发：`SocketComm.cpp:633-698`
- 各协议解析：`SocketComm.cpp:856-1403`

### 12.1 TCP 与 UDP 的帧界差异

UDP 在 `udpsocket.cpp:118-131` 中按 datagram 接收，一次 datagram 就是一条消息。

TCP 服务端和客户端均直接对 `QTcpSocket::readAll()` 的结果发出 `emitNewMessage`：

- `tcpserver.cpp:35-40`
- `tcpsocket.cpp:74-78`

当前没有 TCP 流缓冲、长度字段驱动拆包或多帧循环。因此：

- 文本或固定长度协议通过 TCP 时可能收到半包。
- 两帧可能被合并为一个 `QByteArray`。
- PT2、PT3 对长度做精确判断，TCP 下尤其不可靠。
- 当前 Demo 使用 UDP，可利用 datagram 边界规避。

### 12.2 PT0 JSON

入站是 UTF-8 JSON，没有测数分支。

主要控制字段：

- 数值 `tn=1`：开始。
- 数值 `tn=3`：停止。
- 字符串 `"datapolling"`：轮询模式。
- 字符串 `"dataflow"`：数据流模式。

开始时会检查：

- `bAllOpenCamera`
- `bOnlineCollect`
- `output.iChannelSize`
- `input.iTriggerCtrl`

自动 ACK 为 JSON，字段包括：

```json
{"code":0,"message":null,"tn":1,"xlength":[0],"ylength":[0]}
```

业务出站格式：

```json
{"code":0,"values":[1.23,25.0],"tn":2}
```

由于 PT0 没有测数入站，不存在自动“测数→算法→tn=2”闭环。

### 12.3 PT1 万测

入站只识别配置的三条完整文本：

- `cCMD[0]`：开始
- `cCMD[1]`：停止
- `cCMD[2]`：退出

必须同时满足对应的 `bOpenCMD[]` 和 `input.iTriggerCtrl`。

业务出站把 `vector<double>` 转成分隔文本：

- `iSymbolType=1`：逗号
- `iSymbolType=2`：空格
- `iSymbolType=3`：分号

无测数入站，也无自动 ACK。

### 12.4 PT2 中机双轴

结构体必须保持 1 字节对齐：

```cpp
#pragma pack(push, 1)
struct ControlMessage {
	uint8_t msgType;
	uint8_t command;
};
struct ForceData {
	uint8_t msgType;
	double forceValues[4];
};
struct ServerData {
	uint8_t msgType;
	double valueX;
	double valueY;
};
#pragma pack(pop)
```

固定长度：

- 控制/ACK：2 字节
- 测数：33 字节
- 理论业务结果：17 字节

`double` 直接按 Windows x64 原生 IEEE754 小端布局复制，没有网络字节序转换。对端必须使用相同布局，或显式按小端解码。

控制命令：

- `0x0A`：开始
- `0x0B`：停止
- `0x0C`：轮询/流标志，源码赋值与枚举命名疑似相反
- `0x0D`：流/轮询标志，源码赋值与枚举命名疑似相反
- `0x0E`：查询状态

控制 ACK 的 `msgType=0x03`，由库立即发送。开始 ACK 中存在可疑判断：`bAllOpenCamera==true` 时反而返回 `CAMERA_NOT_OPEN`。

测数 `msgType=0x02` 时，库复制四个 `double` 并发出 `emitNewData(..., 4, 2)`。

#### 业务回发截断缺陷

`CreateZhongJiShuangZhouFour()` 构造 17 字节 `QByteArray`，函数返回类型却是 `QString`：

```509:522:CommHandler/CommSdk/SocketComm.cpp
QString SocketComm::CreateZhongJiShuangZhouFour(const std::vector<double>& data)
{
	if (data.size() != 2)
		return "";
	SocketProtol::ServerData serdata;
	serdata.valueY = serdata.valueX = -1;
	serdata.msgType = SocketProtol::SERVER_DATA_MESSAGE;
	serdata.valueX = data.at(0);
	serdata.valueY = data.at(1);
	QByteArray byteArray(reinterpret_cast<const char*>(&serdata), sizeof(SocketProtol::ServerData));
	return byteArray;
}
```

随后统一调用 `m_controlBox.sendData(QString)`。二进制 `double` 中通常很快出现 `0x00`，隐式文本转换后常只保留首字节 `0x04`。

正确的二进制发送基础设施已经存在：

```cpp
m_controlBox.sendData(byteArray.constData(), byteArray.size());
```

但当前 PT2 未使用。CommLab 因此禁止该业务回发，不能在应用层绕过 DLL。

### 12.5 PT3 三思网口

入站固定 24 字节：

```text
double dForce
double dMove
double dStrain
```

库实际只抛出 `dForce`。

`ParsingProtocolSansi()` 中使用：

```cpp
memcpy(&recv, data.constData(), sizeof(data));
```

形参 `data` 是 `QByteArray`，正确表达应为 `sizeof(Data)` 或 `sizeof(recv)`。当前写法依赖 `QByteArray` 对象大小，属于平台相关缺陷。

没有控制和业务回发分支。

### 12.6 PT4 触发存图

只要收到的字符串中 `\r\n` 出现在索引大于 0 的位置，就发出单次存图事件。

源码使用 `find("\r\n") > 0`，因此当载荷恰好以 `\r\n` 开头时不会触发。MachinePeer 使用 `SAVE\r\n`，满足条件。

### 12.7 PT5 福建威盛

文本格式示例：

```text
TL123.45E67.89D12.34M56.78
```

首字符必须是 `T` 或 `S`。当前只解析 `L` 与 `E` 之间的 load，并抛出一个力值。其余变形、位移、时间字段未进入业务数据。

无控制和回发。

### 12.8 PT6 纳百川自动化

入站 JSON `name`：

- `alphaStartTest`
- `alphaStopTest`
- `alphaSetLD`
- `alphaSetGauge`

后两者通过 `emitEventMsgAndData` 携带标距信息。

业务出站：

```json
{"name":"LVEComputedValues","values":[1.23,25.0]}
```

没有常规测数入站，因此不能自动触发业务结果。

### 12.9 PT7 纳百川线条

入站文本：

- `calcStart`
- `calcEnd`
- `InitYLen:<value>`
- `InitXLen:<value>`
- `InitXYLen:<x>,<y>`

业务出站要求恰好 4 个值：

```json
{
  "hsm": {
    "L1":"...",
    "b1":"...",
    "Le1":"...",
    "bo1":"...",
    "L2":"0",
    "b2":"0"
  }
}
```

### 12.10 网口文本与二进制发送规则

- JSON、万测、纳百川可以使用 `sendData(QString)`。
- 含任意二进制字段的协议必须使用 `sendData(const char*, length)`。
- 不得使用 `strlen` 计算二进制长度。
- 不得把二进制先转成 `QString`。
- UDP 使用 datagram 长度；TCP 必须额外实现流式拆包。

## 13. 电压 AD/DA/CI/DI

### 13.1 模块分层

```text
CommHandler
  → VoltageComm
      → Voltage（VoltageBox）
          → DAQFactory
              ├─ CART_ArtDAQ
              ├─ ART_ARTNET5640
              └─ NI_DAQ
```

`VoltageComm` 构造时就调用 `OpenCtrlBox()`。默认控制器为 ART3134A，因此即使应用只使用串口/网口，进程加载动态库时仍可能需要 `Art_DAQ.dll`。无硬件联调可部署仓库中的 stub。

### 13.2 控制器选择

- `iControllerType=0`：ART3134A
- `iControllerType=1`：NI
- ART5640 工厂类存在，但当前映射分支被注释，正常参数无法选择

### 13.3 AD 采集

配置重点：

- `bADExe[16]`：启用通道
- `dADRange[16][2]`：输入电压范围
- `dValueK[16]`、`dValueB[16]`：线性换算
- `dCompensation[16]`：补偿参数
- `dVADCaliArray[16][10][2]`：10 点标定表

读取链：

```text
CommHandler::ReadADData
  → VoltageComm::ReadADData
      → Voltage::ReadADData
          → ARTBase::ReadDeviceAD
```

实际换算：

```text
result[i] = K[i] × rawVoltage[i] + B[i]
```

当 `bGetVolt=true` 时返回原始电压。

注意：

- `dCompensation` 加法在源码中被注释。
- `dVADCaliArray` 只保存和加载，没有参与读取插值。
- 通过 `setParameter("dValueK"/"dValueB")` 会同步到 `VoltageBox`；只改 `VoltageInfo` 副本可能导致 K/B 不同步。

### 13.4 DA 输出

`SendData(vector, VOLTAGE)` 将业务量映射为电压：

```text
clampedValue = clamp(value, valueMin, valueMax)
voltage = scale × (clampedValue - valueMin) + outputVoltMin
result = clamp(voltage, outputVoltMin, outputVoltMax)
```

然后调用底层 `WriteDAData`。

当前整个向量使用同一个 `iDAChannel` 的缩放参数，不是每个向量元素使用自己的通道参数。

### 13.5 CI 计数器

底层使用 `Dev1/ctr0` 创建边沿计数任务。设备 API 返回 `uInt32`，但 `Voltage::ReadCIFreq` 和公开 API 都返回 `bool`，真实计数值被压缩成真假值。

CI 与硬触发都可能占用 `Dev1/ctr0`，不能同时创建。

### 13.6 DI

当前 ART/NI 实现实际使用 AI `ai15` 作为监听输入，并非典型数字端口读取。

### 13.7 硬触发

```cpp
// 创建指定频率和占空比的连续计数器脉冲。
bool ok = comm.HardTrigger(frequencyHz, dutyCycle);

// 停止并释放硬触发任务。
comm.ReleaseHardTrigger();
```

ART 和 NI 都使用 counter output pulse frequency API。`value` 参数实际是频率 Hz。

### 13.8 电压模块已知缺陷

- CI/DI `Connect()` 没有接收底层返回值，对外可能恒报失败。
- DA `Disconnect()` 用 `dDAVoltRange[i]` 判断通道是否开启，正确字段应为 `bDAExe[i]`。
- ART/NI 部分数组使用 `delete` 而非 `delete[]`。
- NI DA 通道数量初始化与额外任务数组存在风险。
- `emitNewData` 在 VoltageComm 中没有实际发出；AD 是主动轮询读取模式。

## 14. 脉冲模块

### 14.1 硬件类型

- `iPulseCardType=0`：USB AMC2XE
- `iPulseCardType=1`：网络 AMC3XER

主要第三方接口：

```text
OpenUSB_2XE / SOCKET_init
Set_Axs_2XE / Set_Axs
Read_Position_2XE / Read_Position
DeltMov_2XE / DeltMov
CloseUSB_2XE / SOCKET_delete
```

### 14.2 正常输出算法

输入 `vData[0]` 是目标物理量：

```text
quantized = trunc(target / accuracy) × accuracy
diff = quantized - previousQuantized
pulse = trunc(diff / accuracy) × 2
```

随后：

1. 按卡类型确定每秒脉冲上下限。
2. 用 `pulse × frameRate` 判断是否超速。
3. 超速时截断到 `limit / frameRate`。
4. 正数方向为 0，负数方向为 1。
5. 脉冲绝对值若为奇数则加 1，保证偶数。
6. 调用 `DeltMove`。

调用前会忙等 `ReadPos()`，直到 `RunState==0`。该逻辑会阻塞调用线程，不应在 UI 线程高频调用。

### 14.3 标定

```text
pulseCount = dCaliValue / dAccuracy × 2
```

`SendCaliData()` 先发送正方向定长运动，再启动 detached 线程等待停止、清零位置并发出标定完成事件。

### 14.4 默认值

- IP：`192.168.1.30`
- 精度：`0.01`
- 标定值：`0.01`
- 帧率：`20`
- USB：初速 1000，运行速度 200000
- 网络卡：初速 200000，运行速度 2600000

### 14.5 脉冲模块已知缺陷

- 多个 setter/getter 错误地重复使用 `"dFrameRate"` 键，后注册项覆盖前项。
- `SendData` 未检查 `vData.empty()`。
- `UpdatePos` 对第三方返回码的成功判断与 `SendData` 相反。
- 部分事件把 `bPulseOpened` 当成 `ctrlCmd` 参数。
- `PulseInfo` 持有裸 `char*`；二进制保存结构体只保存了指针值。
- 析构函数没有释放 IP 字符数组，也没有确保关闭运动卡。
- detached 标定线程存在对象先析构的生命周期风险。

		[context](void* raw, int count, int type)
		{
			if (!raw || count <= 0 || count > 10000)

			QVector<double> values(count);
			std::memcpy(values.data(), raw, sizeof(double) * count);

## 16. MachinePeer 如何模拟试验机

MachinePeer 的作用不是复用动态库，而是作为独立对端验证动态库。

### 16.1 试验机发给软件

- 串口组包：`MachinePeer/src/SerialProtocol.h::packCommand/packMeasurement`
- 网口组包：`MachinePeer/src/NetworkProtocol.h::packCommand/packMeasurement`
- 串口写入：`MachinePeer/src/MainWindow.cpp::sendCmd/onSendMeasureClicked`
- UDP 写入：`NetworkProtocol::write`

### 16.2 试验机解析软件回包

- 串口：`MachinePeer/src/SerialProtocol.h::parseReply`
- 网口：`MachinePeer/src/NetworkProtocol.h::parseReply`
- 共用结果：`MachinePeer/src/ProtocolResult.h`

日志含义：

- `TX`：试验机发出的真实原始帧
- `RX ACK`：动态库协议级回执
- `RX 解析`：动态库业务回包已被试验机解出
- `RX 未解析`：长度、帧头或字段不匹配

### 16.3 改造真实试验机的步骤

1. 确定物理通道：串口、TCP 或 UDP。
2. 选择动态库已有 PT。
3. 严格按该 PT 入站格式发送，不要参考其他厂商“同名协议”。
4. 控制指令与测数分开发送。
5. 若协议支持回发，按动态库出站格式解析。
6. 区分 ACK 与业务结果。
7. 用十六进制日志或 Wireshark 比对每个字节。
8. 先使用 MachinePeer 验证软件，再用相同黄金帧改造真实试验机。

### 17.5 Stub 的能力边界

`_vendor/ART-DAQ/stub/Art_DAQ_stub.cpp` 只导出加载所需符号，统一返回错误码，不模拟真实 AD、DA、CI 或硬触发。

- 串口/网口联调可以使用 stub。
- 电压硬件功能不能用 stub 验证正确采样。
- 电压功能验证必须换成官方驱动和真实硬件。

## 18. 数据正确性与测试方法

### 18.1 分层验证

第一层：纯组包/解包黄金帧。

- 固定输入值。
- 比对完整十六进制。
- 解包后与原值比较。

第二层：动态库黑盒。

- 通过虚拟串口或 UDP 注入帧。
- 记录 `emitNewData` 次数、长度、类型和值。
- 捕获动态库回包并解码。

第三层：CommLab + MachinePeer 端到端。

- 检查运行门控。
- 检查停止后不回发。
- 检查没有温度的协议不伪造温度。

第四层：真实试验机。

- 使用与 MachinePeer 相同的黄金帧。
- 串口抓取原始字节。
- 网口使用 Wireshark 对照 datagram 长度和 payload。

### 18.2 必测边界

- 空包
- 只有帧头
- 长度不足
- 错误帧头
- 错误 CRC/校验和
- 一个完整帧拆成多次到达
- 多帧一次到达
- 前置垃圾字节后跟合法帧
- `NaN`、`+Inf`、`-Inf`
- 超量程
- 最大/最小整数或浮点值
- 开始前测数
- 停止后测数
- 回发开关关闭
- 连接断开和重复连接

### 18.3 已知黄金帧

三思：

```text
START           = 3D 0D
STOP            = 3E 0D
force=1.0       = E2 00 0F 42 40
```

IEEE：

```text
START           = 24 00 0D 0A
STOP            = 24 FF 0D 0A
```

中机：

```text
START           = 01 0A
STOP            = 01 0B
测数首字节      = 02，后续 4×double 小端
ACK 首字节      = 03
业务结果首字节  = 04，理论总长 17，当前缺陷常为 1
```

### 18.4 数值比较

浮点回环不要直接使用 `==`：

```cpp
// 使用绝对和相对误差共同判断浮点值。
bool nearlyEqual(double actual, double expected)
{
	const double absError = std::fabs(actual - expected);
	const double scale = std::max({1.0, std::fabs(actual), std::fabs(expected)});
	return absError <= 1e-9 * scale;
}
```

文本协议还要考虑格式保留位数造成的量化误差。

## 19. 新增协议的完整路径

### 19.1 先定义协议，不先写 switch

协议说明至少明确：

- 通道和角色
- 帧头
- 版本
- 消息类型
- payload 长度
- 整数和浮点字节序
- 数值单位和缩放
- CRC 或校验和覆盖范围
- 帧尾
- 控制、测数、ACK、业务结果
- 超时和重试
- 最大帧长
- 非法帧处理

推荐二进制帧：

```text
[magic 2B][type 1B][length u16 LE][payload N][crc16 u16 LE][tail 1B]
```

### 19.2 动态库修改点

新增串口协议：

1. `UIDef.h` 为 `iProtocolType` 分配新编号，只追加，不改变旧编号。
2. 在 `Protocol/Serial/` 新建独立编解码器。
3. 在 `SerialPortComm.h` 增加协议对象和接收缓冲。
4. 在 `onRecvDataSlots()` 增加入站分支。
5. 在 `SendData(vector)` 增加业务出站分支。
6. 在 `.vcxproj` 加入新文件。

新增网口协议：

1. 为 `iProtoType` 追加编号。
2. 在 `Protocol/Socket/` 实现协议层。
3. 在 `SocketComm` 增加 TCP 流缓冲。
4. 在 `onRecvDataSlots()` 增加入站分支。
5. 在 `SendData(vector)` 增加业务出站。
6. 二进制必须调用 `sendData(ptr, len)`。

### 19.3 接收状态机

```text
append 新字节
  → 搜索 magic
  → 丢弃 magic 前垃圾
  → 不足固定头：等待
  → 读取 length
  → length 超限：丢弃一个字节并重新同步
  → 不足完整帧：等待
  → 校验 CRC
      ├─ 失败：丢弃当前 magic，继续找下一帧
      └─ 成功：按 type 分发，删除完整帧，继续处理粘包
```

必须给缓冲设置上限，避免攻击或异常数据持续占用内存。

### 19.4 编解码器示例

```cpp
// 计算 Modbus CRC16，返回主机整数值。
std::uint16_t crc16Modbus(const std::uint8_t* data, std::size_t size)
{
	std::uint16_t crc = 0xFFFF;
	for (std::size_t i = 0; i < size; ++i)
	{
		crc ^= data[i];
		for (int bit = 0; bit < 8; ++bit)
			crc = (crc & 1U) ? static_cast<std::uint16_t>((crc >> 1U) ^ 0xA001U)
			                 : static_cast<std::uint16_t>(crc >> 1U);
	}
	return crc;
}

// 按小端顺序追加 16 位无符号整数。
void appendLe16(QByteArray* out, std::uint16_t value)
{
	out->append(static_cast<char>(value & 0xFFU));
	out->append(static_cast<char>((value >> 8U) & 0xFFU));
}

// 构建含长度和 CRC 的完整二进制帧。
QByteArray buildFrame(std::uint8_t type, const QByteArray& payload)
{
	QByteArray frame;
	frame.append(static_cast<char>(0xC0));
	frame.append(static_cast<char>(0xAA));
	frame.append(static_cast<char>(type));
	appendLe16(&frame, static_cast<std::uint16_t>(payload.size()));
	frame.append(payload);
	const auto crc = crc16Modbus(
		reinterpret_cast<const std::uint8_t*>(frame.constData()),
		static_cast<std::size_t>(frame.size()));
	appendLe16(&frame, crc);
	frame.append(static_cast<char>(0x0D));
	return frame;
}
```

### 19.5 入站分发原则

- 控制帧只发事件，不附带虚构测数。
- 测数帧只在完整校验后发 `emitNewData`。
- ACK 由协议层按规范立即发送。
- 业务结果仅由应用算法调用 `SendData` 后产生。
- `emitNewData` 的类型和每个索引含义必须写入协议文档。

### 19.6 CommLab 修改点

- `ProtoGuide.h` 增加下拉项，`itemData` 必须等于库编号。
- `SerialProtocol.h` 或 `NetworkProtocol.h` 增加配置参数。
- 更新 `canReplyToMeasurement`。
- 明确第二通道是温度、状态还是其他量。
- 只有支持闭环的协议才调用 `SendData`。
- 不在 MainWindow 复制协议组包逻辑。

### 19.7 MachinePeer 修改点

- 能力门控：是否可发控制、是否可发测数。
- `packCommand`。
- `packMeasurement`。
- `parseReply`：区分 ACK 和业务结果。
- 串口/TCP 缓冲状态机。
- 原始十六进制日志。

### 19.8 新协议验收标准

新协议只有同时满足以下条件才算“完整可用”：

1. 动态库可配置并连接。
2. 正确解析开始/停止。
3. 正确解析真实测数。
4. 正确处理半包、粘包、垃圾前缀。
5. CRC、长度和最大帧长校验有效。
6. 信号通道数量和语义明确。
7. ACK 可被试验机识别。
8. 业务结果使用正确的文本或二进制发送路径。
9. CommLab 不越过动态库拼包。
10. MachinePeer 可组入站、解出站。
11. 黄金帧、负例和端到端测试全部通过。
12. Debug/Release x64 均能加载正确依赖。

## 20. 源码缺陷与修改优先级

### P0：影响数据正确性

1. 中机 PT2 二进制经 `QString` 截断。
2. `emitNewData` 裸指针生命周期要求未体现在 API 类型中。
3. TCP 没有流式拆包。
4. 科新回包前两个字节未初始化。
5. 三思网口 `sizeof(data)` 写法错误。

### P1：影响功能或稳定性

1. `Disconnect(-1)` 只断开首个模块。
2. 科新半包分支搜索错误帧头。
3. CI/DI Connect 返回值错误。
4. 脉冲参数键重复。
5. 脉冲空向量越界和 detached 线程生命周期。
6. NI/ART 数组释放错误。

### P2：接口和维护问题

1. `SendRequireMsg`、`setForceUnit` 为无效公开 API。
2. 多处注释、协议编号说明与实际分支不同步。
3. 联恒协议编解码器存在，但主分支已注释。
4. 电压标定表和补偿参数未进入实际换算。

修改这些问题时必须同步：

- 动态库源码与头文件
- `CommLab/include/` 的 SDK 头文件副本
- `CommHandler.lib`、`CommHandler.dll`、PDB
- CommLab/MachinePeer 能力门控
- 本手册与 `测试手册.md`

## 21. 快速排错决策树

```text
程序无法启动
  → 检查 CommHandler.dll、Qt DLL、Art_DAQ.dll、脉冲/NI 依赖

Connect=false
  → 检查通道类型、串口索引、IP/端口、Debug/Release、硬件任务

试验机已 TX，CommLab 无 RX
  → 比对 PT 编号和原始字节
  → 检查 input.iTriggerCtrl / iChannelState
  → TCP 检查半包，UDP 检查端口对置

CommLab 有 RX F，无 TX F
  → 检查运行态、回发复选框、bInquireSendFlag
  → 检查该 PT 是否有 SendData 分支

CommLab 有 TX F，Peer 无法解析
  → 检查实际原始长度
  → 区分 ACK 和业务结果
  → 二进制检查是否经过 QString

数值错误
  → 检查字节序、量程、单位、K/B、缩放和文本精度
  → 用黄金帧逐字节复算
```

## 22. 结论

CommHandler 不是一个统一万能协议，而是四个模块和多个固定协议分支的集合。正确使用方式始终是：

```text
按真实协议配置
→ 由动态库接收并解析
→ 应用同步复制数据
→ 应用只处理业务算法
→ 仅在库有对应出站能力时调用 SendData
→ 由动态库组包
→ 试验机按库出站格式解包
```

掌握动态库的关键不只是记住 API，而是同时掌握：

- 每个 PT 的真实能力边界
- 字节级帧格式和数值换算
- Qt 信号中裸指针的所有权
- TCP 流与 UDP datagram 的差异
- 文本路径和二进制路径的区别
- CommLab 与 MachinePeer 各自的职责
- 新协议从库、软件、试验机到测试的完整闭环
