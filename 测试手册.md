# Demo_test 联调测试手册

CommLab 是使用 `CommHandler.dll` 的软件端；MachinePeer 是不使用 DLL 的试验机模拟端。串口与网口互斥，指令与测数分别发送。动态库没有实现的路径不测试、不绕过 DLL。

## 1. 环境

- 两个工程使用 VS2019、Qt `msvc2019_64`、x64。
- 网口默认：CommLab `127.0.0.1:19105`，MachinePeer `127.0.0.1:19106`。
- 串口需要一对虚拟串口，两端参数保持一致。
- 先打开 MachinePeer，再连接 CommLab。

## 2. 动态库能力

### 网口 `iProtoType`

| PT | 协议 | 试验机可发指令 | 试验机可发测数 | 软件业务回发 |
|---:|---|---|---|---|
| 0 | JSON | 开始/停止 | 无 | 有出站分支，但无测数入站触发 |
| 1 | 万测 | START/STOP | 无 | 有出站分支，但无测数入站触发 |
| 2 | 中机 | 开始/停止 | 33 字节 `ForceData` | **不可用：库把 17 字节 `ServerData` 转成 `QString`，实际仅发送 `04`** |
| 3 | 三思 | 无 | 24 字节 `Data` | 无 |
| 4 | 触发存图 | 存图 | 无 | 无 |
| 5 | 福建威盛 | 无 | `TL...E...` 文本 | 无 |
| 6 | 纳百川 | 开始/停止 | 无 | 有出站分支，但无测数入站触发 |
| 7 | 纳百川线条 | 开始/停止 | 无 | 有出站分支，但无测数入站触发 |

中机控制 ACK 使用库内部二进制发送接口，仍可收到 2 字节回执。只有 `SendData(vector)` 的中机业务结果路径受 `QString` 截断缺陷影响；在不改动态库的约束下无法恢复力温。

### 串口 `iProtocolType`

| PT | 协议 | 试验机可发指令 | 试验机可发测数 | 自动业务闭环 |
|---:|---|---|---|---|
| 0 | 三思 | 开始/停止/存图/清零 | `E2` + 4 字节力 | 是，仅回真实入站力 |
| 1 | 科新 | 无控制事件 | `02 4C` + 9 字节 ASCII 力 | 是，仅力 |
| 2 | 时代新材 | 无 | 四字段 CSV + `\r\n` | 否，库 `SendData` 为空 |
| 3 | IEEE | 开始/停止 | 无 | 否；只有独立出站分支 |
| 4 | 冠腾 | 无 | 无 | 否；只有独立 `R#N#` 出站分支 |

## 3. 必测流程

### 串口三思完整闭环

1. 两端选择串口 PT0 并打开。
2. MachinePeer 点“开始”，CommLab 应显示 `RX CMD 开始`。
3. MachinePeer 点“发力温”，CommLab 应显示 `RX F=` 和 `TX F=`。
4. MachinePeer 应收到并解析仅力回包；输入温度不会被伪造回发，因为三思入站帧没有温度。
5. 点“停止”后再发测数，CommLab 可以收数但不得业务回发。

### 串口科新

1. 两端选择串口 PT1 并打开；该协议没有开始指令，CommLab 连接后直接允许测数。
2. MachinePeer 发测数，TX 必须以 `024C` 开头。
3. CommLab 收到力后调用库 `SendData`；MachinePeer 只显示力，不显示伪温度。

### 网口中机

1. 两端选择网口 PT2，MachinePeer 打开 UDP，CommLab 连接。
2. 点“开始”，验证 CommLab 控制事件和 MachinePeer 2 字节 ACK。
3. 点“发力温”，CommLab 应显示四通道入站中的前两个值。
4. CommLab 明确记录中机业务回发不可用，不调用 `SendData(vector)`，因此不再产生误导性的单字节 `04`。

### 其余协议

- 灰色按钮表示动态库无对应入站能力。
- 网口 PT3/PT5 与串口 PT2 只验证软件接收，不期待业务回发。
- 网口 PT0/PT1/PT6/PT7 只验证控制事件；库虽然有出站分支，但当前交互没有测数来源，不由指令伪造业务结果。
- 网口 PT4 只验证存图事件。

## 4. 日志判定

- `RX CMD`：动态库已解析控制指令。
- `RX F=`：动态库已解析测数并抛出 `emitNewData`。
- `TX F=`：CommLab 已调用动态库 `SendData(vector)`。
- `RX ACK`：协议回执，不是算法结果。
- `RX 解析`：MachinePeer 已按动态库出站格式解出业务数值。
- `INFO`：动态库能力边界，程序按设计跳过。
- `ERR`：打开、组包或解析失败。

## 5. 通过标准

- 串口三思、科新闭环通过。
- 指令不会连带发送测数或业务结果。
- 没有真实温度的协议不显示、不回发伪温度。
- 中机测数入站和控制 ACK 正常，业务回发明确标记为动态库缺陷。
- 串口与网口不能同时连接，协议按钮与能力表一致。

代码入口：CommLab 的 `SerialProtocol.h` / `NetworkProtocol.h`，MachinePeer 的同名两个协议适配头文件。
