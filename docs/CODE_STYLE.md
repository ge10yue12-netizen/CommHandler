> 文档版本：V1.0

> 稳定路径：`docs/CODE_STYLE.md`（版本号写在文内，不写在文件名）  
> 适用范围：CommunicationAssistant生产代码、测试代码及适配层  
> 上级约束：最新正式《开发基线》

## 1. 文档定位

本文件统一规定：

1. 代码如何编写；
2. 注释如何编写；
3. 修改者如何完成自检和自测。

本文件不重新定义以下内容：

- 产品范围和里程碑；
- Session、Transport、Capture、Scheduler和Legacy的架构归属；
- 公共接口与持久化格式；
- 缺陷严重度；
- PASS、CONDITIONAL PASS和FAIL判定。

以上内容分别由最新开发基线和代码变更审计规范负责。发生冲突时，执行顺序为：

```text
最新开发基线
→ 代码编写/注释规范（本文第一部分、第二部分）
→ 自检自测规范（本文第三部分）
→ 代码审计规范
→ 单次变更对齐单
```

说明：

- 本文件为单一稳定入口；冲突裁决时「编写/注释」高于「自检自测」，二者均高于审计规范中的流程与证据要求表述（若表述冲突）。
- **文件改动门禁口令「确认修改」与最终 PASS/CONDITIONAL PASS/FAIL 判定仍以《代码变更审计规范》为准**；本文件不授权绕过门禁，也不自行给出最终审计结论。
- 单次任务对齐单不得覆盖上级文档。

本文件取代以下旧文件，旧文件不再作为执行依据：

```text
CommunicationAssistant_CODE_STYLE_COMMENTS_SELFTEST_V1.0.md
CommunicationAssistant_CODE_WRITING_STANDARD_V1.0.md
CommunicationAssistant_CODE_COMMENT_STANDARD_V1.0.md
CommunicationAssistant_CODE_SELF_CHECK_TEST_STANDARD_V1.0.md
```

# 第一部分：代码编写规范

## 2. 适用边界

### 2.1 新工程代码

CommunicationAssistant所有新增和实际修改的代码执行本规范。

### 2.2 遗留代码

对于开发基线声明冻结或仅兼容使用的遗留代码：

- 不批量格式化；
- 不为了风格一致制造大面积Diff；
- 不修改任务无关命名；
- 不复制后未经验证直接使用；
- 是否允许修改以开发基线和变更任务为准。

## 3. 工具配置

项目根目录应提供：

```text
.clang-format
.clang-tidy
.editorconfig
```

| 文件            | 作用                                      |
| --------------- | ----------------------------------------- |
| `.clang-format` | 统一缩进、换行、大括号和Include等机械格式 |
| `.clang-tidy`   | 检查常见缺陷、现代C++、性能和可读性问题   |
| `.editorconfig` | 统一UTF-8、换行、缩进和行尾空格           |

工具版本和实际命令必须在本地环境验证后写入工程说明。不得虚构工具可用性或检查结果。

只格式化本次修改的新工程文件，禁止对无关文件执行全仓格式化。

## 4. 文件与目录

- 文件位置服从最新开发基线的模块划分；
- 一个主要类型通常对应同名`.h/.cpp`；
- 测试文件使用`TypeNameTest.cpp`；
- 公共基础类型放在基线指定的核心模块；
- 下层模块不得反向依赖UI；
- 不建立当前里程碑不需要的空框架；
- 不使用`Utils2`、`Temp`、`CommonAll`等模糊名称；
- 文件名应体现主要职责。

## 5. 命名

```cpp
class NativeSession;                 // 类型：PascalCase
struct SessionConfig;                // 结构体：PascalCase
enum class SessionState;             // 枚举类型：PascalCase
SessionState::Connected;             // 枚举值：PascalCase

Result configureSession();           // 函数：lowerCamelCase
QString localAddress;                // 参数/局部变量：lowerCamelCase
QString localAddress_;               // 私有成员：尾部下划线

constexpr int kMaximumQueueSize = 1000;
```

规则：

- 禁止匈牙利命名；
- 禁止无意义缩写；
- ID、IP、TCP、UDP等通用缩写可以使用；
- 同一概念只使用一个名称；
- 布尔名称表达谓词，如`isConnected`、`canSend`；
- 函数名表达动作，禁止`handle1()`、`process2()`；
- 类型名表达角色，避免无边界的`ManagerAll`、`GlobalHelper`。

## 6. 格式

- UTF-8，无BOM；
- 4空格缩进，禁止Tab；
- 单行建议不超过120字符；
- 文件末尾保留换行；
- 禁止行尾空格；
- 一个声明一行；
- 大括号、换行和Include排序以`.clang-format`为准；
- 指针和引用靠近类型：

```cpp
QObject* parent;
const QByteArray& bytes;
```

Include建议分组：

```cpp
#include "NativeSession.h"

#include <memory>
#include <utility>

#include <QByteArray>
#include <QPointer>

#include "core/CommRecord.h"
#include "transport/ITransport.h"
```

机械格式由工具统一，不在代码审查中反复争论个人风格。

## 7. 类型与常量

- 状态、模式、错误和协议编号使用`enum class`；
- 使用`nullptr`，禁止`NULL`；
- 禁止用宏定义普通常量；
- 禁止魔法数字；
- 时间、长度、大小和频率使用明确单位；
- 字节长度和元素数量不得混用；
- 外部输入缩窄转换前必须校验范围；
- 二进制数据使用`QByteArray`，禁止用`QString`承载任意二进制。

```cpp
constexpr qsizetype kMaximumFrameBytes = 1024 * 1024;
constexpr int kOpenTimeoutMs = 5000;
```

## 8. 所有权与生命周期

- 优先RAII；
- 禁止新增手工成对`new/delete`；
- 独占所有权使用`std::unique_ptr`；
- 使用共享所有权必须说明必要性；
- QObject非所有权引用可使用`QPointer`；
- 公共接口避免无语义`void*`；
- 跨线程传递的数据必须拥有稳定生命周期；
- 不保存仅在回调期间有效的指针；
- 对象在正确线程创建和销毁；
- 关闭后迟到事件必须能够安全处理。

## 9. 参数、返回值与错误

- 小型标量按值传递；
- 不修改的大对象使用`const&`；
- 可能失败的业务操作返回明确结果类型；
- 不用模糊`bool`吞掉错误原因；
- 输出参数必须有充分理由；
- 错误包含操作、对象和底层原因；
- 异常不得跨DLL边界；
- 禁止空`catch`；
- 不静默忽略连接、发送、解析和写盘失败。

## 10. 控制流与状态

- 优先早返回，减少深层嵌套；
- 非法状态必须显式拒绝；
- `switch`处理未知值；
- 禁止无限等待；
- 禁止无限重试；
- 禁止无界队列；
- 禁止用固定`sleep`代替可靠同步；
- 异步请求接受、提交、完成和失败必须区分；
- 状态只有一个权威修改者。

## 11. Qt规则

- UI对象只在UI线程访问；
- QObject在所属线程创建和销毁；
- 使用新式函数指针`connect()`；
- `DirectConnection`只用于明确需要的即时调用；
- Direct槽中禁止UI更新、文件写入和复杂解析；
- UI槽中禁止阻塞I/O；
- 不为每个周期任务创建线程；
- 不用大量UI `QTimer`替代统一调度；
- `deleteLater()`只能在对象线程事件循环有效时使用；
- 跨线程信号的数据类型和生命周期必须安全。

## 12. 通信代码通用规则

以下是实现质量规则，不替代开发基线中的具体模型：

- 不依赖TCP单次read边界；
- UDP保持Datagram边界；
- TCP Server不同客户端状态必须隔离；
- 本地发送状态不得描述成远端确认；
- 原始字节和解析结果不得混为同一概念；
- 发送、记录和异步事件使用基线规定的关联标识；
- 协议长度、端序和校验未经证据验证时不得猜测；
- 所有队列必须有容量、溢出策略和错误记录；
- 网络、串口和文件错误必须保留系统错误信息。

## 13. 禁止事项

禁止：

- 无边界巨型类；
- UI直接执行通信细节；
- 使用协议显示名称判断能力；
- 把一次read默认当作协议帧；
- 使用QString保存二进制；
- 后台线程直接更新UI；
- 空错误提示；
- 没有任务依据的顺手重构；
- 为假设中的未来需求提前搭建框架；
- 复制遗留代码后未经测试直接宣称可复用。

# 第二部分：代码注释规范

## 14. 注释目标

注释解释代码无法直接表达的原因、约束和风险，不复述代码，不承担测试证明责任。

注释重点回答：

- 为什么这样设计；
- 线程和生命周期限制是什么；
- 异步结果在什么条件下成立；
- 协议规则的依据是什么；
- 哪些内容是暂时假设；
- 修改此处可能破坏什么。

## 15. 语言与术语

- 标识符使用英文；
- 注释以简洁中文为主；
- 协议正式字段名保留原文；
- 单位使用`ms`、`ns`、`Hz`、`bytes`等标准缩写；
- 术语服从最新开发基线；
- 同一概念不得使用多个名称。

## 16. 事实标记

涉及协议、遗留行为和外部系统时使用：

| 标记           | 含义                             |
| -------------- | -------------------------------- |
| `[VERIFIED]`   | 已通过源码、文档、测试或设备验证 |
| `[DECISION]`   | 已冻结决定，引用基线位置         |
| `[ASSUMPTION]` | 临时假设，注明验证方法           |
| `[BLOCKED]`    | 缺少证据，禁止继续推断           |
| `[DEFERRED]`   | 不属于当前里程碑                 |

```cpp
// [VERIFIED] 长度字段表示Payload字节数，小端。
// 证据：协议文档V1.2第4.3节与Golden样本case-007。
```

```cpp
// [ASSUMPTION] 当前按CRLF提取文本消息。
// 验证方法：真实设备逐字节分片、连续多帧和正文换行测试。
```

## 17. 必须注释的内容

### 公共接口

说明功能、同步/异步、参数单位、返回语义、线程要求和主要错误。

```cpp
/**
 * @brief 请求打开会话。
 *
 * 返回成功仅表示请求已被接受。最终状态通过状态事件报告。
 *
 * @thread Application线程调用，不执行阻塞I/O。
 */
Result open() override;
```

### 线程

说明对象在哪个线程创建、调用和销毁，信号从哪个线程产生，是否允许阻塞或重入。

### 所有权

说明裸指针、临时缓冲区、非所有权引用和跨线程数据的有效期。

```cpp
// data仅在回调期间有效，必须在返回前复制。
// 禁止保存指针或通过QueuedConnection继续转发。
```

### 异步语义

说明请求被接受、本地提交、本地完成和远端确认之间的区别，具体状态名称引用开发基线。

### 协议规则

说明帧边界、长度含义、端序、校验范围、字段单位和证据来源。完整协议结构优先放专项协议文档，代码只保留必要说明和引用。

### 单位与范围

```cpp
qint64 timeoutMs;        // ms，允许范围1～60000
qint64 monotonicNs;      // 单调时钟ns，不表示UTC时间
qsizetype payloadBytes;  // 字节数，不是元素数量
```

### 非直观兼容逻辑

解释为何不能采用看似简单的实现，以及兼容限制带来的风险。

## 18. TODO、FIXME和BLOCKED

所有待办必须关联变更编号：

```cpp
// TODO(CA-M2-LEGACY-003):
// 验证遗留回调元素类型后再开放对应能力。
```

```cpp
// FIXME(CA-BUG-TCP-004):
// 客户端断开与广播并发时需要保证Channel快照一致。
```

```cpp
// BLOCKED(CA-M4-CODEC-002):
// 缺少正式协议文档，不能确定长度字段含义。
```

禁止无编号的“以后优化”“临时处理”。

## 19. 禁止的注释

禁止：

- 复述代码；
- 大段注释掉的旧实现；
- 人工维护文件修改历史；
- 没有证据的“已经验证”；
- “绝对安全”“不可能失败”等无限结论；
- 与当前实现不一致的过期注释；
- 用注释掩盖错误命名和复杂设计。

```cpp
++index; // 禁止：只是复述代码
```

历史由Git、变更编号和审计报告维护。

## 20. 测试注释

测试注释用于说明：

- 特殊数据来源；
- Golden报文来源；
- 为什么某种分片能够复现Bug；
- 固定随机种子的原因；
- 不明显的同步条件。

测试名称已经清楚时不重复描述测试步骤。

## 21. 注释维护

- 修改逻辑时同步检查相邻注释；
- 删除已经失效的注释；
- 架构说明放开发基线，不复制到多个源文件；
- 注释引用最新文档版本或规则编号；
- 注释不是测试证据。

# 第三部分：代码自检自测规范

## 22. 自检、自测与审计

| 环节 | 责任                           | 输出         |
| ---- | ------------------------------ | ------------ |
| 自检 | 修改者检查范围、Diff和静态问题 | 自检记录     |
| 自测 | 修改者执行构建和测试           | 自检自测报告 |
| 审计 | 按审计规范核查证据与基线       | 最终审计结论 |

自检自测只能决定“是否可提交审计”，不能自行给出最终PASS。

## 23. 开始条件

开始修改前必须具有：

- 变更编号；
- 最新开发基线版本；
- 变更对齐单；
- 计划修改文件；
- 明确验收标准；
- 计划构建和测试方式；
- 已识别的现有修改；
- 已知假设和BLOCKED项。

缺少这些内容时只允许只读检查。

## 24. 自检分层

### L0：范围

- 修改是否仍在计划文件内；
- 是否覆盖已有修改；
- 是否突破里程碑或基线；
- 是否产生无关格式化；
- 是否把Bug修复扩大成重构。

### L1：静态检查

- 格式；
- Diff空白；
- 编译器警告；
- 静态分析；
- 空指针、生命周期和整数转换；
- 线程亲和性；
- 无界资源和永久等待；
- 状态和错误路径。

### L2：构建

- 配置；
- 编译；
- 链接；
- Qt MOC和资源；
- Debug构建；
- 任务影响Release时增加Release构建；
- 新增警告。

未实际执行必须写“未构建”。

### L3：自动测试

- 相关单元测试；
- 模块集成测试；
- 回归测试；
- 错误路径和边界输入；
- 时间、并发和随机分片测试。

### L4：本机I/O

- 本机TCP；
- 多客户端；
- UDP双端；
- 虚拟串口；
- 端口占用；
- 主动断开。

### L5：设备与稳定性

- 真实设备；
- 拔插与断网；
- 长时间运行；
- 高速数据；
- 磁盘不足；
- 实际用户流程。

每次任务记录实际达到的最高层级和未覆盖风险。

## 25. 修改前后Diff

Git工程修改前至少记录：

```bash
git status --short
git diff --name-only
git diff --stat
```

修改后至少执行或等效检查：

```bash
git status --short
git diff --check
git diff --stat
git diff --name-only
```

逐文件阅读实际Diff，检查计划外文件、无关格式化、调试代码、临时文件、敏感信息和遗漏测试。

非Git工程必须说明替代比较方式。

## 26. 测试编写规则

- 测试名称表达条件和预期结果；
- 一个测试验证一个主要行为；
- 测试可重复，不依赖执行顺序；
- 不共享可变全局状态；
- 失败信息包含预期和实际值；
- 时间逻辑优先注入FakeClock；
- 不使用不必要的真实`sleep`；
- 随机测试固定并记录种子；
- 文件测试使用临时目录；
- 网络测试优先使用临时端口；
- Golden数据记录来源和版本；
- 测试代码执行本规范的代码与注释要求。

示例：

```text
configure_rejectsInvalidConfiguration
open_reportsAcceptedBeforeConnected
send_preservesCorrelationId
scheduler_submitsNextItemAfterConfiguredInterval
tcpServer_keepsClientChannelsIndependent
capture_preservesRawBytesExactly
```

## 27. 证据等级

| 等级 | 证据               | 能证明的范围             |
| ---- | ------------------ | ------------------------ |
| E0   | 代码阅读           | 只能发现明显问题         |
| E1   | 构建成功           | 语法、类型和链接基本成立 |
| E2   | 单元测试           | 隔离逻辑在测试输入下成立 |
| E3   | 集成测试           | 模块组合路径成立         |
| E4   | 本机真实I/O        | 本机通信路径成立         |
| E5   | 真实设备或长期测试 | 目标设备和稳定性场景成立 |

禁止用低等级证据替代高等级验收，例如用编译成功证明协议正确，或用Loopback证明跨机器网络正常。

## 28. 通用必测类别

每次修改根据涉及模块选择对应测试，不在本文重新定义架构预期；具体预期引用开发基线或需求编号。

### Core与Session

- 合法与非法配置；
- 状态迁移和非法状态；
- 异步请求语义；
- 关联标识贯穿；
- 关闭和迟到事件。

### Resource冲突

- 标识规范化；
- 通配绑定；
- 内部冲突；
- 系统调用失败；
- 地址复用边界。

### Serial

- 参数校验；
- 打开、关闭、占用和拔出；
- 二进制零字节；
- 读写错误；
- 无永久等待。

### TCP Client

- 连接、拒绝、超时和断开；
- 多次读取顺序；
- 本地发送状态语义；
- 资源释放。

### TCP Server

- 多客户端隔离；
- 定向发送和广播；
- 单客户端断开；
- Channel生命周期；
- 并发边界。

### UDP

- Datagram边界；
- 源和目标端点；
- 空Datagram；
- 绑定失败；
- 地址复用。

### Scheduler

- 单次、次数和无限任务；
- 暂停、恢复、停止；
- 顺序轮询；
- 周期锚点；
- 失败策略；
- FakeClock。

### Capture

- 原始字节往返一致；
- 格式字段稳定；
- 队列满和写失败；
- 会话隔离；
- 记录类型不混淆；
- 异常文件识别。

### Legacy

- 线程创建和销毁；
- 临时指针立即复制；
- 输入边界；
- 能力限制；
- 超时和不可响应；
- 不伪造原始数据。

### Simulator与Codec

- 故障规则可重复；
- 固定随机种子；
- Golden正常与异常数据；
- 随机分片和粘包；
- 错误后恢复；
- 缺少协议依据时停止实现。

## 32. AI执行要求

AI完成修改后必须：

1. 重新读取实际Diff；
2. 检查计划外文件；
3. 运行当前环境实际可用的格式、构建和测试；
4. 明确未运行项；
5. 不编造命令和输出；
6. 不把代码阅读描述成测试通过；
7. 不把一次运行扩大为长期稳定结论；
8. 生成自检自测报告；
9. 提交独立审计；
10. 审计前只写“可提交审计”，不自行判定最终PASS。
