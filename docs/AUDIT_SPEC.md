# CommunicationAssistant 代码变更审计规范

> 文档版本：V1.2  
> 文档性质：**强制执行规范**（与开发基线配套，不可互相替代）  
> 稳定路径：`docs/AUDIT_SPEC.md`  
> 配套基线：`docs/BASELINE.md`（以仓库内正式发布的最新版本为准；当前 V1.2.4）  
> 适用范围：CommunicationAssistant 的设计、编码、重构、Bug 修复、测试与文档变更  
> 门禁：即使用户已确认任务意图，**仍须用户明确回复「确认修改」后才允许改文件**；本规范不取消该门禁

---

## 0. 文档体系与门禁

| 文档 | 作用 |
|---|---|
| `docs/BASELINE.md` | 冻结产品范围、架构边界、接口语义、里程碑与工具链 |
| `docs/CODE_STYLE.md` | 代码编写、注释、修改者自检自测 |
| `docs/AUDIT_SPEC.md`（本文） | 管理每次变更编号、对齐、证据、测试、偏差与验收结论；**文件改动门禁与最终 PASS 判定** |

文档冲突裁决顺序（与基线 / CODE_STYLE 一致）：

```text
最新开发基线
→ 代码编写/注释规范（CODE_STYLE 第一、二部分）
→ 自检自测规范（CODE_STYLE 第三部分）
→ 代码审计规范（本文）
→ 单次变更对齐单
```

标准开工指令：

```text
请先读取 docs/BASELINE.md、docs/CODE_STYLE.md 与 docs/AUDIT_SPEC.md，
创建变更编号并填写变更对齐单；
在用户回复「确认修改」并确认范围后再实施修改；
完成后按 CODE_STYLE 完成自检自测并生成自检自测报告，
再生成代码变更审计报告；
没有构建或测试证据的项目不得标记为通过。
```

执行顺序（强制）：

```text
用户提出任务
→ 只读：读基线 + CODE_STYLE + 本规范 + 相关源码
→ 输出变更对齐单（零文件改动）
→ 用户回复「确认修改」（可附带范围说明）
→ 实施修改
→ 按 CODE_STYLE 自检 / 构建 / 测试并生成自检自测报告
→ 按本文生成《代码变更审计报告》
→ 审计结论 PASS / CONDITIONAL PASS / FAIL
```

**无「确认修改」口令 → 零文件改动**（含工程创建、删除、批量格式化）。  
口头「可以 / 开始吧 / 做吧」等不构成授权。

以后所有 AI/开发交付必须同时对齐基线、CODE_STYLE 与本规范；仅口头「已完成」视为无效交付。

## 1. 为什么需要独立审计规范

开发基线回答：系统应该做什么，哪些边界不能突破。  
本规范回答：一次修改为什么发生、修改了什么、是否越界、如何验证、是否允许进入主线。

没有审计记录的代码修改，视为未完成，不得仅凭口头说明认定为可交付。

---

## 2. 强制原则

1. **无变更编号，不修改代码。**
2. **无修改前对齐单，不开始跨文件编码。**
3. **无用户「确认修改」，不改任何文件。**
4. **无文件清单，不允许扩大修改范围。**
5. **无测试证据，不得声称功能完成。**
6. **未运行的测试必须明确写出，不能省略。**
7. **不能编译时必须写「未编译」，不得用静态阅读代替构建成功。**
8. **修改超出基线时必须先变更基线，不能先写代码后补理由。**
9. **不得把推测、示例输出或 AI 生成结论当成真实运行结果。**
10. **不得顺手重构与任务无关的代码。**
11. **用户已有修改不得被覆盖、回退或格式化掉。**
12. **每次审计必须能从需求追踪到代码，再从代码追踪到测试。**
13. **审计不通过时不能以「基本可用」代替问题处理。**

## 3. 变更编号

```text
CA-M1-CORE-001
CA-M1-SERIAL-001
CA-M1-TCP-001
CA-M1-SCHED-001
CA-M2-LEGACY-001
CA-M3-SIM-001

CA-BUG-模块-三位序号
```

格式：`CA-里程碑或类型-模块-三位序号`  
同一任务多轮修改沿用同一编号，不得每次修补新建编号割裂历史。

文档/基线/本规范自身变更可用：`CA-DOC-*`。

## 4. 事实可信度

| 标记 | 含义 |
|---|---|
| `[VERIFIED]` | 已通过源码、构建、测试或设备验证 |
| `[DECISION]` | 已冻结的产品或架构决定 |
| `[ASSUMPTION]` | 临时假设，必须说明验证方法 |
| `[UNVERIFIED]` | 尚未核读，不得当事实或可复用承诺 |
| `[BLOCKED]` | 缺少必要证据，不能安全继续该项 |
| `[DEFERRED]` | 明确不属于当前任务或里程碑 |

不得作为审计证据：

- 「代码看起来没问题」；
- 「理论上能够运行」；
- 「应该已经修复」；
- 「AI 判断没有风险」；
- 「和常见项目写法一样」；
- 没有命令、日志、测试名或设备记录的「测试通过」。

## 5. 一次变更的完整生命周期

```text
提出任务
→ 修改前对齐（只读）
→ 用户「确认修改」
→ 只读检查工作区 / Diff 基线
→ 按清单实施修改
→ 自检与构建
→ 单元 / 集成 / 设备测试
→ 检查实际 Diff
→ 生成审计报告
→ 审计结论
→ 允许合并或退回修改
```

任一步发现基线冲突、未知协议事实、用户修改覆盖风险或无法验证的关键行为，必须停止扩大修改并报告。

以下情况必须再次请求确认后才能继续：

- 修改文件超出对齐单；
- 需要删除或重命名公共接口；
- 需要改变配置或抓包格式；
- 需要修改动态库；
- 需要增加当前里程碑之外的模块；
- 需要覆盖无法判断归属的现有修改；
- 任务从 Bug 修复演变为架构重构。

## 7. 修改范围审计

### 7.1 修改前

Git 工程至少检查：

```bash
git status --short
git diff --name-only
git diff --stat
```

目的：识别用户已有修改；区分本次与历史变更；避免覆盖无关文件；防止把已有代码误报为本次成果。

### 7.2 修改后

```bash
git status --short
git diff --check
git diff --stat
git diff --name-only
```

并逐文件检查实际 Diff。禁止只看最终内容不看 Diff。

若非 Git 工程，须用文件清单/备份比较，并在报告中声明无法使用 Git 审计。

## 8. 需求—代码—测试追踪矩阵

| 需求编号 | 需求/问题 | 实现文件 | 关键接口 | 验证方式 | 结果 |
|---|---|---|---|---|---|
| R1 | … | `src/...` | `Class::method` | `TestName` | 通过/失败/未运行 |

规则：一需求可对多文件/测试；每个修改文件须对应需求或必要构建变更；P0/P1 须有验证方式；无需求映射=范围漂移；无测试映射的关键逻辑=未验证。

## 9. 通用代码审计清单

### 9.1 正确性

输入边界；空/超长/非法枚举；非法状态拒绝；错误上下文；是否只走正常路径；是否把异步请求接受误认为完成；溢出/截断；未初始化；错误生命周期或悬空引用。

### 9.2 线程与异步

UI 是否阻塞 I/O；QObject 线程亲和性；跨线程数据生命周期；DirectConnection 是否仅用于立即复制；回调中是否写盘/UI/重计算；超时是否被说成取消成功；关闭后迟到事件；状态是否单一权威写入者。

### 9.3 资源

有界队列；无限循环/重试/等待；可靠关闭；异常路径释放；重复打开；TCP Server 每客户端独立；UI 模型是否无界堆积。

### 9.4 可维护性

无关抽象；重复逻辑；枚举语义稳定；接口所有权清晰；注释写原因；禁止靠协议名字符串判断能力；临时兼容不扩散到核心层。

## 10. 模块专项审计（对齐 BASELINE V1.2+）

### 10.1 Core 与 Session

- `configure()` 仅 `Created/Closed`；强类型 `SessionConfig`；
- `ResourceClaim` 由 `claimFor` 推导，无双重真相；
- `open/close/send` 异步请求语义；`open()` 无连接参数；
- UI 只依赖 `ICommSession`；
- `requestId`/`taskId`/`sessionId`/`channelId` 贯穿事件（顶层字段）。

### 10.2 Serial

COM 规范化；参数合法；占用/拔出/读写错误；`RawChunk` 非 Frame；无永久阻塞；关闭后迟到信号安全。

### 10.3 TCP Client

连接超时与取消区分；断线状态事件；RawChunk 顺序；不依赖单次 read 边界；本地提交≠对端收到；未授权不偷偷重连。

### 10.4 TCP Server

每客户端唯一 `channelId`；状态/统计独立；选中/广播记录 channelId；断开回收；通配冲突；**M1 不得加 Codec 缓存框架**。

### 10.5 UDP

保留 Datagram 边界；记录源/目标；区分固定目标与按来源回复；`addressReuse` 仅尝试；不按 TCP 流处理。

### 10.6 SendScheduler

仅 M1 冻结范围；周期锚点 **Submitted**；稳定 `requestId`；Failed 默认停任务；暂停/停止语义正确；统一调度器，禁止每任务线程或大量 UI Timer。

### 10.7 Capture

CaptureManager → 每 Session 独立 Writer/队列/文件；格式为 **JSON Lines（`.clog.jsonl`）**；枚举英文字符串、UUID 小写无花括号、64 位与纳秒十进制字符串、`bytesBase64`；顶层字段不被 `attributes` 覆盖；UI 清空不影响文件；写失败隔离并出 ErrorEvent；Legacy 不冒充 Raw；无虚假回放入口。

### 10.8 Legacy（M2）

Worker 内创建/销毁 CommHandler；Direct 槽只检查+立即复制；Voltage/Pulse 未验证保持关闭；仅一个活动 LegacySession；Watchdog 在 Worker 外；Unresponsive 拒新命令；不向阻塞 Worker 发 close；不声称有 DLL 未公开原始字节。

### 10.9 Simulator / Codec（M3/M4）

Raw 故障不冒充 CRC；可重复种子；Codec 有 Golden；支持分片/粘包测试；缺文档则 BLOCKED；Simulator ≠ 固定 TCP Server。

## 11. 测试证据等级

| 等级 | 证据 | 可证明什么 |
|---|---|---|
| E0 | 代码阅读 | 仅明显问题，不证明运行正确 |
| E1 | 编译成功 | 语法/类型/链接基本成立 |
| E2 | 单元测试 | 隔离逻辑在给定输入下成立 |
| E3 | 集成测试 | 模块组合路径成立 |
| E4 | 本机真实 I/O | 串口/Socket 等系统路径成立 |
| E5 | 真实设备/长期 | 目标场景与稳定性 |

禁止用低等级证据替代高等级验收。AI 审查 ≠ 运行时线程安全证明。

---

## 12. 测试结果记录

每条至少：测试名称、命令/步骤、环境、输入、预期、实际、通过/失败/未运行、日志位置。

缺设备时写明未运行原因、当前证据等级、交付为条件通过或不通过。

---

## 13. 缺陷严重度

| 等级 | 定义 | 处理 |
|---|---|---|
| P0 | 崩溃、损坏、死锁、严重越界、核心不可用、破坏基线 | 必须修，审计不通过 |
| P1 | 功能/状态错误、泄漏、显著误导、关键验收缺失 | 原则上修后再通过 |
| P2 | 可维护性、提示、次要边界 | 可条件通过并记录 |
| P3 | 格式命名等 | 可后续任务 |

不得靠降低严重度规避修复。

## 14. 审计结论（仅三选一）

**PASS**：范围与基线一致；无未处理 P0/P1；必要构建与测试已执行；证据足以支持验收。

**CONDITIONAL PASS**：无 P0；剩余问题不影响当前核心路径；未验证项与后续动作已写明；不得描述成完全完成。

**FAIL**：存在 P0 或影响验收的 P1；范围失控；未批准突破基线；无法区分本次与已有修改；声称测试通过无证据；关键路径未编译或未按要求验证。

## 17. 里程碑审计（任务通过 ≠ 里程碑通过）

**M1**：四 Transport 真实可用；RawChunk/UdpDatagram 语义；TCP Server Channel；Scheduler 冻结范围与 Submitted；Capture JSONL；未链 CommHandler；UI 不直接依赖 Transport；资源冲突与错误可验证。

**M2**：DLL 未改；线程亲和性；Direct 立即复制；能力表二元保守；Unresponsive 不提交；Legacy 不冒充 Raw。

**M3**：模拟器独立于 DLL；故障可重复；波形经统一 Scheduler；Legacy 波形受能力限制。

**M4**：Codec 有协议证据与 Golden；可重解析；缺文档协议保持 BLOCKED。

```text
开发基线：决定能做什么、不能做什么
```
